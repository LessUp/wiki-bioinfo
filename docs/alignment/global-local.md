# 全局比对与局部比对

## 是什么

序列比对并不是只有一种“对齐方式”。在最经典的教学框架里，至少要区分两类目标：

- **全局比对（global alignment）**：尽量对齐两条序列的整体；
- **局部比对（local alignment）**：只寻找最相似的局部片段。

这个区分非常重要，因为它决定了算法在边界处理、递推关系和生物学解释上的差异。

## 为什么重要

不同数据场景会提出完全不同的问题：

- 比较两个长度接近的同源基因时，我们往往关心整体差异；
- 在数据库中搜索保守结构域时，我们往往只关心局部高相似片段；
- 对一个 read 去寻找参考中的候选位置时，也更接近局部匹配而不是整串完全对齐。

因此，“到底该做全局还是局部”不是实现细节，而是问题建模本身的一部分。

## 核心概念

### 全局比对

全局比对默认希望两个序列从头到尾都参与比对。经典模型是 Needleman–Wunsch。

它的边界条件要求“从左上角走到右下角”，意味着序列两端也要被解释为匹配、替换或 gap。

常见递推可写为：

$$
  F(i,j) = \max
  \begin{cases}
    F(i-1,j-1) + s(x_i, y_j) \\
    F(i-1,j) + g \\
    F(i,j-1) + g
  \end{cases}
$$

其中 `s(x_i, y_j)` 为匹配/替换分数，`g` 为 gap 罚分。

### 局部比对

局部比对允许只取两个序列中的一段高分片段进行对齐。经典模型是 Smith–Waterman。

<figure>
  <img src="/wiki-bioinfo/img/illustrations/smith-waterman.svg" alt="Smith-Waterman 局部比对示意图" />
  <figcaption>局部比对允许在得分掉到 0 时重新开始，因此更适合寻找保守片段、motif 和局部同源区域。</figcaption>
</figure>

局部比对最大的区别是：它允许“从任意位置开始，也在任意位置结束”。因此递推里通常会多出一个 0：

$$
  H(i,j) = \max
  \begin{cases}
    0 \\
    H(i-1,j-1) + s(x_i, y_j) \\
    H(i-1,j) + g \\
    H(i,j-1) + g
  \end{cases}
$$

这个额外的 0 很关键，它表示：如果当前前缀对齐已经不值得继续，就直接从这里重新开始。

### 什么时候用哪一种

**全局比对**更适合：

- 两条序列长度接近；
- 预期整体同源；
- 希望比较全长差异。

**局部比对**更适合：

- 序列长度差异较大；
- 只关心某个保守结构域或 motif；
- 希望在大型数据库中找最相似片段；
- 一条序列可能只与另一条序列局部同源。

## worked example

考虑两条序列：

- `x = TCGA`
- `y = ACG`

如果做全局比对，算法会尝试解释两条序列整体的关系，即使首尾不匹配也要纳入最终答案。

如果做局部比对，算法更可能聚焦于中间保守片段 `CG` 或 `ACG` 的高分区域，而忽略无关的边缘字符。

这说明：

- 全局比对强调“整体一致性”；
- 局部比对强调“局部高质量信号”。

## 与真实工具或流程的连接

在最基础的动态规划实现里，两者时间复杂度通常都是：

- 时间复杂度：$O(mn)$
- 空间复杂度：$O(mn)$

差异主要不在复杂度，而在：

- 边界条件；
- 回溯终点；
- 得分解释；
- 生物学任务适配性。

现实工具中：

- **Needleman–Wunsch** 常作为教学中的全局比对代表；
- **Smith–Waterman** 常作为局部比对的精确代表；
- **BLAST** 更接近“先找局部高相似 seed，再扩展”；
- 短读长 mapping 工具通常结合索引结构和局部精细比对，而不是直接做完整二维 DP。

## 常见误区

- 局部比对一定比全局比对更先进；
- 全局比对只适用于完全相同的序列；
- 局部比对找到的片段就是唯一正确的生物学答案；
- 两者的差别只是算法写法不同，与任务无关。
