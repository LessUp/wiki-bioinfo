# 参考版本、构建号与坐标转换

## 是什么

在基因组分析中，“参考基因组版本”和“坐标系”是所有结果的基础：

- 人类基因组：GRCh37、GRCh38、hg19、hg38 等；
- 其他物种也有自己的构建号和版本体系；
- 不同版本之间可能存在插入、缺失、重排以及命名差异。

**liftover** 则是指：

- 把一个版本上的坐标（如某个变异的位置）映射到另一个版本上；
- 使用预先构建的链文件（chain files）和比对关系来实现。

## 为什么重要

如果参考版本搞混：

- 同一个坐标在不同版本上可能对应不同碱基；
- 注释文件与 BAM/VCF 不匹配，会导致 feature 定位错误；
- 与数据库对照时，版本不一致会造成“找不到记录”或误解释结果。

## 典型场景

1. 变异结果要与公共数据库对照：
   - 数据库使用 GRCh37 还是 GRCh38？
   - 你的 VCF 是基于哪个版本的参考（通常写在 header 里）。

2. 不同研究之间结果整合：
   - 一份数据在 hg19，另一份在 hg38；
   - 想做 meta 分析或交集时，需要先统一坐标系。

3. 历史数据和新数据共存：
   - 很多旧资源仍在 GRCh37/hg19 上；
   - 新分析逐渐迁移到 GRCh38。

## 如何确认参考版本

- 查看 BAM/CRAM header 中的 `@SQ` 与 `@HD` 信息；
- 查看 VCF header 中的 `##reference=` 字段；
- 查看注释文件（GTF/GFF）的 metadata 或下载页面；
- 仔细阅读数据库或资源的文档说明。

如果不能确认版本，建议：

- 先用几个已知变异或基因位置在不同版本上比对，检验是否一致；
- 谨慎起见，宁可先查清楚来源再做大规模分析。

## liftover 的基本思路

liftover 依赖的是：

- 版本之间的全基因组比对（alignment）；
- 从而得到一系列“区间对齐关系”（chain 文件）；

在使用时：

- 输入：旧版本上的坐标（如 chr1:123456）；
- 查找：对应链文件中与之重叠的区间；
- 输出：新版本上对应的坐标（或提示无法映射）。

## 实践建议

- liftover 前：确认源版本和目标版本；
- liftover 后：
  - 检查是否有大量 unmapped/失败的位点；
  - 检查几个关键位点是否仍处于预期基因或区域中；
- 如果新的分析都在新版本上，尽量把旧结果统一 liftover 过去，而不是反向适配。

## 与注释和数据库的关系

- 注释文件（GTF/GFF）和数据库记录也都有版本；
- 当前变异坐标的版本应与注释和数据库版本一致；
- 公开资源通常会明确标注 “GRCh37-based” 或 “GRCh38-based”等信息。

## 注意事项

- 版本号（GRCh37 vs hg19）有时包含细微差异，要阅读对应文档；
- liftover 并不完美，特别是在结构复杂区域；
- 不要在同一项目中混用不同版本而不做记录。
