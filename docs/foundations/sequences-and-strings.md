# 序列、字符串与坐标系统

## 是什么

在计算机视角下，DNA、RNA 和蛋白质序列都可以看作有限字母表上的字符串：

- DNA：`{A, C, G, T}`
- RNA：`{A, C, G, U}`
- 蛋白质：20 个氨基酸字符

这种抽象并不是形式上的简化，而是把生物信息学问题转成可以计算的问题：我们可以讨论模式匹配、编辑距离、动态规划、索引结构、压缩表示，以及区间坐标与注释映射。

## 为什么重要

一旦把序列抽象成字符串，就可以自然引入：

- 子串与模式匹配；
- 编辑距离与近似匹配；
- 动态规划与比对；
- suffix array、BWT、FM-index 等索引结构；
- k-mer 计数、压缩表示和概率模型。

但真实分析并不只处理“字符串本身”，还要同时处理**坐标系统**。很多错误不是出在算法，而是出在“这段序列在参考上的哪个位置、这个区间到底怎么定义、链方向该如何解释”这些基础问题上。

## 核心概念

### 序列是对象表示，字符串是计算抽象

在生物学里，DNA、RNA 和蛋白质是不同层次的对象；在算法里，它们都可以先表示成字符序列。这样做的意义在于：

- 生物学问题可以落到有限字母表上的计算问题；
- 很多算法能够跨对象层复用；
- “read、motif、gene sequence、protein domain sequence” 虽然语义不同，但都可以先以字符串方式处理。

### 子串、模式与近似匹配

很多核心任务都可以重新表述为“模式在序列里如何出现”：

- 一个 primer 或 motif 是否在序列中出现；
- 一个 read 是否能在参考基因组中找到兼容位置；
- 一个序列与另一个序列是否只差少量替换、插入、缺失。

因此，精确匹配和近似匹配是后续比对、索引和定量问题的起点。

### 坐标系统决定字符串结果如何落回生物对象

除了序列本身，还要理解区间坐标：

- 起点、终点如何定义；
- 区间是 closed 还是 half-open；
- 不同文件格式是否采用 0-based 或 1-based；
- 正负链方向如何影响 feature 的解释；
- 跨 exon 的 transcript 坐标如何映射回基因组坐标。

这些看似细节的问题，在注释、可视化和流程串联时经常引起系统性错误。

## worked example

假设一个 read 比对到参考基因组上的某个外显子附近，想判断它是否支持某个 transcript：

1. 先把 read 当作字符串，判断它与参考上哪些位置兼容；
2. 再看这些位置是否跨越剪接位点；
3. 再把基因组坐标映射到 transcript 坐标；
4. 最后结合注释判断它支持的是哪个 exon 或哪个 isoform。

这个例子说明：字符串匹配只回答“能不能对上”，坐标系统和注释对象才回答“它在生物学上意味着什么”。

## 与真实工具或流程的连接

- **序列比对**：把 read 或 query sequence 作为模式，在参考中寻找精确或近似匹配；
- **序列索引**：为了加速搜索，构建 k-mer、suffix array、BWT、FM-index 等结构；
- **RNA-seq 定量**：需要同时理解序列兼容关系与 transcript 坐标层级；
- **注释解释**：variant 或 feature 的意义依赖它落在什么区间、什么链方向、什么注释对象上；
- **文件格式转换**：BED、GFF/GTF、VCF、BAM 等常因为坐标约定不同而出现 off-by-one 错误。

## 常见误区

- 认为序列问题只要会做字符串匹配就够了，而忽略坐标和注释语境；
- 混淆 0-based 与 1-based 坐标；
- 混淆 closed interval 与 half-open interval；
- 忽略正负链，导致对 feature 方向和转录本结构解释错误；
- 把 transcript 坐标和 genome 坐标直接混用；
- 看到一个匹配位置就默认结果唯一，忽略重复序列和 multi-mapping。
