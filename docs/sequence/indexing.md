# 索引结构概览

## 是什么

索引结构的核心任务，是让我们不必每次都从头到尾扫描整条参考序列，就能快速定位某个模式可能出现的位置。

如果把生物序列看作长文本，那么索引就是“为了查询而预处理的额外结构”。它常见于：

- read mapping；
- 数据库搜索；
- motif 扫描；
- 大规模同源序列检索。

## 为什么重要

面对大规模参考基因组或海量数据库序列时，逐位置扫描通常代价过高。更现实的问题是：

- 怎样快速找到候选匹配区域？
- 怎样减少无意义的二维动态规划？
- 怎样在内存、速度和敏感性之间做平衡？

现代序列工具之所以能在大数据上工作，很大程度上依赖索引结构而不是单纯依赖更快的 CPU。

## 核心概念

### 哈希表与 k-mer 索引

最直观的办法是：

- 从参考序列提取 k-mer；
- 用哈希表记录每个 k-mer 出现的位置；
- 查询时先找共享 seed，再决定是否继续扩展。

优点是简单直接，缺点是内存开销较大，对错误和重复也比较敏感。

### 后缀数组 / 后缀树

后缀结构把“所有后缀”组织起来，使得模式匹配可以更系统地完成：

- 后缀树理论表达更直观；
- 后缀数组更适合工程实现与压缩；
- 它们帮助我们理解为什么字符串查询能被大幅加速。

### BWT 与 FM-index

BWT / FM-index 是生物信息学中极其重要的一类压缩索引思想。

它的价值在于：

- 保留高效查找能力；
- 同时降低空间消耗；
- 非常适合大规模参考序列查询。

### minimizer 与稀疏 seed

在长读长或超大数据库场景下，往往不会保留所有 k-mer，而是选取代表性的稀疏 seed，例如 minimizer。

这样可以降低：

- 存储开销；
- 比较次数；
- 查询过程中的噪声负担。

## worked example

假设参考序列中已经建立了 4-mer 索引。查询 read 时：

1. 提取 read 中若干 4-mer；
2. 在索引中查找这些 4-mer 出现的位置；
3. 如果多个 seed 落在参考的相邻区域，就把该区域作为候选位置；
4. 最后再对候选区域做更精细的局部比对。

这个流程说明：索引的目的不是直接给出最终答案，而是快速缩小搜索空间。

## 与真实工具或流程的连接

现代比对工具往往并不是纯动态规划，而是：

1. 先用索引找到候选区域；
2. 再对局部区域做精细比对；
3. 最终综合得分与过滤条件输出结果。

所以从工程视角看，索引负责“快”，动态规划负责“准”。

不同索引通常在以下维度上做权衡：

| 方案 | 优点 | 代价 |
|---|---|---|
| k-mer 哈希 | 简单直观，查询快 | 内存大，对重复敏感 |
| 后缀数组 / 树 | 字符串理论清晰 | 工程复杂度较高 |
| FM-index | 压缩效果好，适合大参考 | 理解和实现更复杂 |
| minimizer | 更稀疏，适合长读长 | 可能损失一部分敏感性 |

## 常见误区

- 索引结构只是为了提速，与结果质量无关；
- 只要索引够好，就不需要动态规划；
- 所有读长和所有任务都适合同一种索引；
- 索引建立后查询就不会再受到重复序列影响。
