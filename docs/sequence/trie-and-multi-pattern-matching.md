# Trie 与多模式匹配

## 是什么

如果我们只有一个模式串，可以讨论单模式匹配；

但在很多场景下，我们要找的是：

- 一组 motif；
- 一组 k-mer；
- 一组 primer / barcode；
- 一批已知保守模式。

这时问题变成：

> 如何在一段长文本中，同时寻找很多模式串？

**Trie**（字典树）是一种非常自然的表示方式：

- 每条从根到叶的路径表示一个模式串；
- 共享前缀的模式会共用路径；
- 这样就能把“多个模式”组织到一个统一结构里。

## 为什么 trie 有用

假设要匹配这些模式：

```text
AT
ATA
ATC
CG
```

如果单独对每个模式做扫描，会有很多重复工作，特别是：

- `AT`、`ATA`、`ATC` 共享前缀 `AT`；
- 在文本扫描时，不该每次都把这个前缀重复处理。

Trie 正好把这类共享前缀结构显式表示出来。

## 一个直观例子

构造 trie 后：

- 根节点分出 `A` 和 `C` 两条边；
- `A` 之后接 `T`；
- 从 `AT` 又分出 `A` 和 `C`；
- `C` 之后接 `G`；

于是：

- 多个模式共享前缀；
- 在文本中扫描时，只要沿 trie 向下走，就能同时检查多条模式。

## 多模式匹配为什么难

如果文本很长，而模式很多：

- 对每个模式各扫一遍，代价会很高；
- 模式之间往往有前缀共享；
- 失配后如果总回到根开始，会浪费很多已有信息。

因此，更高级的多模式匹配算法（如 Aho–Corasick）会进一步引入：

- trie 结构；
- failure links / fallback links；
- 在单次文本扫描中同时处理所有模式。

## Aho–Corasick 的直觉

Aho–Corasick 可以理解为：

- 先把所有模式建成 trie；
- 再为 trie 的节点补充“失配时应该跳到哪里”的 failure link；
- 扫描文本时，不必每次失配后都回到根，而可以跳到另一个合适前缀状态继续。

这让它具有两个重要特征：

- 文本通常只需要扫一遍；
- 可以同时报告多个模式的出现位置。

## 在生物信息学中的位置

Trie / 多模式匹配对生物字符串问题很自然，因为：

- motif 集合常常不是一个，而是一组；
- k-mer、barcode、adapter 等往往需要批量检测；
- 很多模式共享前缀；
- 在大规模序列数据库或长 reads 中，一次扫完整个文本更高效。

## 与 suffix 结构的区别

- **Trie / Aho–Corasick**：
  - 更适合“很多模式去扫一个文本”；
- **Suffix tree / suffix array / FM-index**：
  - 更适合“一个大文本被预处理后，反复回答很多查询”；

这两类结构都很重要，但解决问题的方向不同。

## 常见误区

### Trie 只是字符串教材里的玩具结构

不是。它提供了多模式匹配、prefix sharing 和失败回退的最自然直觉。

### 多模式匹配只是把单模式匹配重复很多次

不对。真正高效的方法会利用模式之间的共享结构，而不是机械重复。
