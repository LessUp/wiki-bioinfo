title: "如何贡献"
---


欢迎通过 Issue 和 Pull Request 参与建设 BioInfo Wiki。

这个项目希望像 OI Wiki 一样，形成一套**可持续协作、结构清晰、术语一致、页面互相联结**的中文知识库，而不仅是若干孤立文章。

## 适合贡献的内容

- 新知识点页面
- 对现有页面的纠错、补图、补例子
- 对术语、公式、图示的统一整理
- 对学习路线、交叉链接和 landing page 的改进
- 对站点样式、Docusaurus 组件和自动化检查的优化
- 对工作流、数据格式、数据库资源等桥接性页面的完善

## 推荐贡献流程

1. 先开 Issue，或在已有 Issue 下说明想认领的任务。
2. Fork 仓库并创建分支。
3. 修改文档、插画或站点代码。
4. 本地运行：

```bash
npm install
npm run start
npm run check
```

5. 在 Pull Request 中说明：
   - 修改目标
   - 影响页面
   - 是否新增了图片 / 组件 / 目录结构
   - 你是如何验证的

## 内容贡献的优先方向

当前最欢迎的方向包括：

- 基础桥接页：reads、coverage、参考基因组、注释、文件格式
- 算法主线页：索引、比对、组装、概率模型
- 工作流页：DNA-seq、RNA-seq、metagenomics、注释流程
- 数据资源页：数据库入口、版本选择、对象映射
- 可视化增强：流程图、对比表、符号说明、术语卡片

## 写文档时建议注意

- 优先解释概念之间的关系，不要只堆术语；
- 先说明问题，再说明方法，不要直接堆命令；
- 图示、公式、表格都应服务于正文解释；
- 尽量说明方法的前提、优点和局限；
- 能给出站内交叉链接时尽量给出；
- 对不确定或仍待考据的内容请明确标注。

## 内容质量标准

一篇合格页面至少应尽量满足：

- 有清晰的主题和读者目标；
- 有基本定义与背景；
- 有关键模型、流程或任务解释；
- 有例子、图示、表格或 worked example；
- 有参考资料；
- 有相关页面链接；
- 能放进站点整体知识图谱中，而不是孤立存在。

## 图片与组件贡献

如果你要补充插图、流程图或 MDX 组件，请优先保证：

- 视觉元素能直接解释问题，而不是纯装饰；
- 命名与目录结构清晰；
- 暗色模式下也可读；
- 组件 API 尽量简单，适合后续文档复用。

## 修改前最好同步的页面

在写新页面前，建议先阅读：

- [项目简介](./index.md)
- [学习路线](./roadmap.md)
- [写作规范](./style-guide.md)

这能帮助你把新内容写到正确的层级里，也能减少术语和结构风格的不一致。
